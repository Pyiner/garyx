Garyx Implemented Design

绑定 Channel 的 StreamEvent 扇出设计

当前实现把线程运行产生的 committed StreamEvent，按线程绑定关系统一扇出到所有 channel 端点；内置 channel 和新外部 plugin 使用同一个 StreamDispatchEnvelope outbound 契约，gateway/router 不承担 Markdown、图片、工具调用或平台特殊规则的渲染职责。

核心边界 Gateway/router 只做事件和端点选择；channel 自己消费 StreamEvent 并渲染。
主路径 内置 channel 和新外部 plugin 都消费 StreamDispatchEnvelope。
兼容路径 旧 subprocess plugin 才由 adapter 降级到 dispatch_outbound(ChannelOutboundContent)。

1. 目标架构

API/gateway 入口在已有稳定 thread_id 后直接 snapshot；channel/plugin inbound 入口先用 DeferredBoundStreamFanout 缓冲事件，等 router 解析出 canonical thread_id 后，通过 DeferredFanoutAgentDispatcher 在 provider dispatch 前 snapshot、排除 origin、flush 缓冲。route_and_dispatch 返回后的 attach 只是幂等兜底。内部入口只是 run 来源，不是另一套 outbound 协议。

统一 StreamEvent 扇出

run 来源无关，channel 渲染内聚在 garyx-channels

2. Run 生命周期和绑定变更

run 开始时 snapshot target set。运行中修改 bot/channel 绑定不会影响当前 run；streaming input 追加到同一个 run 时也不会重新选目标。新绑定从下一次 run 开始生效。

Snapshot 语义

稳定性优先，避免一个 run 中途改变投递集合

3. Endpoint Identity 和去重

这里的边界收窄为：gateway/router 只认 channel 给出的 opaque Endpoint Identity。它不关心这个 key 怎么生成、里面有什么字段，也不解析任何 channel 内部字段。

只按 Endpoint Identity 做 equality

生成和语义归属在 channel，gateway 只比较 key 是否相等

4. Dispatcher Contract 和数据

主路径 callback 接收 StreamDispatchEnvelope。新 subprocess plugin 的 JSON-RPC DispatchStreamEvent 由这个 envelope 直接转换出来，避免内部和外部插件协议字段漂移。当前主路径不伪造 per-target sequence；如果后续需要 replay recovery，sequence 必须来自 committed per-run replay。

字段 含义 归属边界
endpoint_identity 端点唯一 key，用于去重、origin 排除、outbound id 关联 channel 生成；gateway/router 只做 equality
thread_id / run_id canonical thread 和当前 run fanout snapshot 和 committed replay 使用
account_id / chat_id channel 账号和主要投递目标 channel renderer 使用，gateway 不解析
delivery_target_type / delivery_target_id typed target，例如 chat_id/open_id 等 channel 自己解释
delivery_thread_id 平台 thread/topic/scope channel 自己解释；router 只保存关联
event 原始 StreamEvent，包含 Delta、Boundary、ToolUse、ToolResult、Done 等 保持结构化直到 channel renderer
当前内置 native callback：Telegram、Feishu、Discord、Weixin。Feishu 的非 origin 目标没有原始 Feishu message id，因此无法把 COT thread 挂回用户那条消息，但仍然收到完整 StreamEvent，由 Feishu renderer 自己退化为普通卡片/图片。

5. Plugin 协议和 Legacy Adapter

新外部 plugin 和内置 channel 对齐同一套 outbound 事件语义：目标消费 StreamDispatchEnvelope。差异只在传输形态，内置 channel 是 Rust callback，新 subprocess plugin 是 host -> plugin 的 dispatch_stream_event RPC。唯一允许不同的是旧 subprocess plugin adapter，它只是兼容旧协议的临时路径。

host 只通过注册或初始化 capability 识别新旧协议：声明 capabilities.dispatch_stream_event = true 的 plugin 走 native RPC；没有声明的旧 plugin 才走 adapter。

插件协议选择

新插件对齐 StreamEvent，旧插件走临时兼容 adapter
目标类型 协议选择 工具调用语义
内置 channel dispatcher callback 接收 StreamDispatchEnvelope 保持结构化，交给 channel renderer
新 subprocess plugin dispatch_stream_event(StreamDispatchEnvelope) 与内置 channel 同语义
旧 subprocess plugin adapter 输出 dispatch_outbound(ChannelOutboundContent) ToolUse / ToolResult 走旧协议已有结构化类型

6. 当前代码落点和验收

代码落点

  1. garyx-channels/src/dispatcher.rs：统一 envelope、native callback、legacy adapter fallback。
  2. garyx-channels/src/bound_fanout.rs：deferred fanout、provider-dispatch 前挂载 wrapper、origin endpoint 排除、缓冲 flush。
  3. garyx-channels/src/plugin_host/*：外部 plugin capability 和 dispatch_stream_event RPC。
  4. garyx-gateway/src/application/chat/delivery.rs：gateway/API 入口 bound delivery。
  5. garyx/src/channel_plugin_host.rs 和内置 channel handlers：入口 unknown thread 时使用 deferred fanout。

验收重点

  • run 来源无关，所有绑定端点都能收到 committed event。
  • gateway/router 只按 Endpoint Identity equality 去重。
  • 结构化 tool event 不在 gateway 降级。
  • 新 plugin 和内置 channel 对齐 StreamDispatchEnvelope 语义。
  • 旧 plugin adapter 保留 ToolUse / ToolResult 结构化透传。
  • 内置 Feishu、Discord、Weixin、Telegram 都有 native stream-event callback。
  • outbound message id 继续回写 router。
  • deferred fanout 在 provider dispatch 前 snapshot，运行中绑定变更不进入当前 run。
  • 当前不伪造 event seq；restart recovery / per-event diagnostics 后续补 typed outcome。

7. 已落地和剩余债务

已落地

  • 内置 channel 和新外部 plugin 都消费同一套 stream-event envelope。
  • 新旧 plugin 用 dispatch_stream_event capability 识别。
  • 旧 plugin 走 dispatch_outbound(ChannelOutboundContent) adapter。
  • 运行中修改绑定不影响当前 run，下一次 run 生效。

剩余债务

TODO旧 subprocess plugin adapter 只为兼容旧协议保留；所有插件升级到 dispatch_stream_event 后删除。

Diagnostics当前可观测性以 outbound message id persistence 和日志为主；独立 per-event delivery outcome 可后续补。
